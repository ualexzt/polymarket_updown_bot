# Polymarket BTC UP/DOWN State-Pricing Bot (PAPER)

A PAPER-only state-pricing bot for Polymarket BTC 5m/15m UP/DOWN markets.

The bot is a **value-entry decision engine**:
1. Pulls the current round from the Polymarket Gamma API.
2. Pulls recent 5m candles from Binance BTCUSDT.
3. Computes the round state (current side, distance bucket, volatility bucket, candle pattern, stage).
4. Looks up a historical UP/DOWN probability from a pre-computed state-bucket rules table.
5. Compares `fair_price = historical_probability` against the live Polymarket ask, with a safety buffer.
6. Issues a TRADE only if `ask <= max_buy_price` and all other filters pass; otherwise SKIP.
7. PAPER-executes at the best ask (no partial fills, one position per (market, side)).
8. Records mark-to-market snapshots while the position is open.
9. Settles on resolution (Polymarket API primary, Binance fallback).

v1 has **no live trading**. All trades are simulated.

## Quickstart

```bash
# 1. Install
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

# 2. Build the state rules (one-time; from the research CSV)
python scripts/build_state_rules.py

# 3. Configure
cp .env.example .env
# Edit .env if you want to override defaults (paper mode is on by default)

# 4. Run a single decision cycle against an explicit URL
python scripts/run_polymarket_round_paper.py \
    --event-url "https://polymarket.com/uk/event/btc-updown-15m-1780652400" \
    --mode paper --once

# 5. Or run continuously against the currently active 15m market
python scripts/run_polymarket_round_paper.py --timeframe 15m --mode paper

# 6. Inspect results
python scripts/paper_report.py
python scripts/export_paper_trades.py --out paper_trades.csv
python scripts/inspect_paper_trade.py --position-id pos_xxxxxxxxxxxx
```

## Docker

A `Dockerfile` and `docker-compose.yml` are provided for headless
paper runs (e.g. on a server). The bot runs as a non-root user, in
UTC, with `restart: unless-stopped`.

```bash
# 1. Build the image
docker compose build

# 2. Smoke test: one decision cycle against an explicit URL
docker compose run --rm bot \
    --event-url "https://polymarket.com/uk/event/btc-updown-15m-1780731900" \
    --mode paper --once

# 3. Start continuous paper mode (15m timeframe, auto-discovery)
docker compose up -d
docker compose logs -f

# 4. Inspect on the host (DB and logs are bind-mounted)
python scripts/paper_report.py
```

### Notes
- The `data/`, `logs/`, and `config/` directories are bind-mounted
  from the host, so the SQLite DB, logs, and state rules persist
  across container restarts.
- The container runs as a non-root user matching the host user
  (`HOST_UID`/`HOST_GID` env vars, default `1000:1000`). Override if
  your host user is different:
  `HOST_UID=$(id -u) HOST_GID=$(id -g) docker compose up -d`.
- The `STATE_RULES_PATH` and `DATABASE_PATH` are set in the
  `Dockerfile` to absolute `/app/...` paths. All other settings come
  from `.env` on the host (loaded via `env_file:`).
- `TZ=UTC` is hardcoded in `docker-compose.yml` — Polymarket rounds
  are anchored to UTC `:00/:15/:30/:45` boundaries, so do not change
  this.
- Logs are rotated by the Docker json-file driver
  (`max-size: 10m`, `max-file: 3`). For multi-day runs, also tail
  the SQLite `decisions` table via `scripts/paper_report.py` for
  the decision funnel (markets seen / rule lookups / trade conditions
  / TRADE).

## State Rules

The rules file is generated from the research CSV at:

```
/home/alex/Project/poly_bot_system/out_rounds_v2/BTCUSDT_5m_180d_state_bucket_report.csv
```

The CSV contains 2014 rules (17273 rounds, 180 days). After filtering for
`usable_signal=True` AND `return_aligned=True` AND `samples >= 60` AND
`historical_probability >= 0.60`, **58 strong rules** remain. The remaining
rules are still kept in the file so the engine can fall back to them.

Lookup hierarchy (in order):
1. Exact: `stage + current_side + distance_bucket + volatility_bucket + pattern`
2. Fallback: `stage + current_side + distance_bucket + pattern` (drops volatility)
3. Fallback: `stage + current_side + distance_bucket` (drops pattern)
4. No match → SKIP

No-trade conditions (always enforced):
- `samples < MIN_SAMPLES` (60)
- `historical_probability < MIN_HISTORICAL_PROBABILITY` (0.60)
- `return_aligned != true`

## Config

See `.env.example` for the full list. Key parameters:

| Setting | Default | Description |
|---|---|---|
| `BOT_MODE` | `paper` | v1 only supports `paper` |
| `BTC_SYMBOL` | `BTCUSDT` | Binance symbol |
| `SAFETY_BUFFER` | `0.04` | subtract from `fair_price` to get `max_buy_price` |
| `MIN_EDGE` | `0.04` | minimum `fair_price - ask` for TRADE |
| `MAX_SPREAD` | `0.03` | maximum bid-ask spread for TRADE |
| `MIN_LIQUIDITY_USD` | `25` | minimum top-5 depth for TRADE |
| `MIN_HISTORICAL_PROBABILITY` | `0.60` | minimum rule probability for TRADE |
| `MIN_SAMPLES` | `60` | minimum rule sample count for TRADE |
| `MAX_POSITION_USD` | `5` | PAPER position size |
| `MAX_OPEN_POSITIONS` | `1` | global open-position cap |
| `MAX_DAILY_LOSS_USD` | `10` | daily realised PnL floor |

## 5m vs 15m markets

- **15m markets**: full AFTER_5M and AFTER_10M state machine. Rules in
  `config/btc_updown_state_rules_15m.json` apply.
- **5m markets**: `CUSTOM_5M_STATE` (no internal candle segmentation).
  The current rules table is for 15m only, so 5m markets will SKIP with
  `no_rule_for_state` until 5m-specific rules are generated. This is
  intentional and audit-friendly — we never trade without a calibrated
  rule.

## Why this bot is honest

Every TRADE and every SKIP is persisted with a full snapshot:

- BTC state (round open, current price, current side, distance bucket,
  volatility bucket, candle pattern, stage)
- Polymarket orderbook (best bid, best ask, spread, top-5 depth, age)
- Polymarket metadata (slug, event slug, condition id, UP/DOWN token
  ids, accepting orders, age)
- Rule (rule id, match type, samples, historical probability,
  return_aligned)
- Risk (requested size, max position, open count, daily PnL)

If a SKIP happens, the reason is recorded. If a TRADE happens, the
exact ask, max_buy_price, and edge are recorded. After settlement, the
trade quality (GOOD_WIN / BAD_WIN / GOOD_LOSS / BAD_LOSS / EXECUTION_ERROR)
is classified for review.
